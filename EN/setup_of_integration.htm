Setup of integration

 
Before starting settings
1. Define a testing order. Perhaps, it will be required to unroll a system on the selected test server. To avoid distortion of data in a product system when debugging and testing integration.
 
2. Resolve safety issues. Systems can exchange the sensitive information including a trade secret of the enterprise. For example, finance, technologies, client orders, etc.
 
3. Procedures of audit and diagnostics. In case of problems, the administrator should establish quickly the reason, whenever possible, repeatedly to execute exchange on a data set after elimination of a problem.
It is desirable to have diagnostic aids and monitoring of productivity of exchange. Means of the notification in case of problems.
 
 
The system gives ability to integrate with adjacent systems by means of web api and XML files. 
By means of any data web api can be imported and exported are available in the graphic user interface. 
 
Note: Application within the project web api потребуется the help of specialists of implementation.
 
Most we will stay in the way for mastering of integration exchange of XML files is.

System settings. Integration
 

Import of data.

By XML exchange it is possible to fill reference books of a system:
  •  Load reference texts of control programs;
  •  Specify the list of the made products;
  •  Give the description of technology of their production.
  •  Load the periods of routine maintenance and planned daily tasks.
 
For such exchange four directories are used.
IN – the directory where incoming requests are located. DPA periodically browses contents of this directory, and processes all files appearing in this directory.
DONE – all successfully processed files automatically will be moved to this directory.
ERROR – will move those files when which processing there was an error to this DPA directory. At the same time the text of an error and stacktrace will be added to a file body. It helps at diagnostics of problems and when debugging.
OUT – will place responses to incoming requests in this DPA directory.
Integration. Import
 
In setup only one root way for import is specified. It is supposed that in this root directory there are 4 subdirectories – IN, DONE, ERROR, OUT.
"Import is activated" – to process incoming requests or not.
"Create responses to external requests" – to create answers in the OUT directory or not.
 
Export data
DPA поддерживает экспорт трех категорий данных:
1) fact of execution of tasks (operations, adjustments, M&R, implementations). Including the actual start time and end, quantity (suitable / conditionally suitable / defects), a workplace, the operator, information on suspensions, etc.
2) marks about the equipment downtime reasons with detailing on each interval of downtime
3) the fact of execution of cycles unitary enterprise, including the modes, the errors CNC and other information
 
Each category of export has separate setup
Example. Export data
 
Directories which are specified in settings of import and export can be directories of local file system of the server, or it can be network folders on the general network resource. In the second case the path will be like \\fileserver\DPA\Import\Tasks.
 
Together with a system the xsd-schemes allowing the services accompanying the systems which are already implemented at the enterprise quickly are delivered to master a format of unloading of data.
 
Exchange by MSMQ
DPA supports export of data through MSMQ queues. Setup happens similar to export through the file system, only instead of a way to the directory it is necessary to specify URL of queue of MSMQ. It is possible to use local or remote queues.
Make sure that the rights for access (record in queue) are configured correctly.
